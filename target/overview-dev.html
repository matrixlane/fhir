FHIR概述 - 开发者

[%wgt fhir%] 工作组成熟度级别: N/A标准状态:信息性

FHIR（Fast Healthcare Interoperability Resources）旨在支持各种环境下的医疗信息交换，以支持医疗保健的提供。该规范基于并调整了现代广泛使用的RESTful实践，以实现跨各种团队和组织的综合医疗保健提供。

FHIR的预期范围广泛，涵盖人类和兽医、临床护理、公共卫生、临床试验、管理和财务方面。该标准旨在全球范围内使用，并适用于各种架构和场景。

框架

FHIR基于“资源”，它们是所有交换的共同构建块。资源是某种医疗实体的实例级表示。 所有资源在通用方面具有以下特点：

  • 资源的标识符 - 通常是定义资源位置的URL
  • 通用元数据
  • 可读的XHTML摘要
  • 一组定义的数据元素 - 每种资源类型都有不同的一组
  • 可扩展性框架，以支持医疗保健的变化

资源实例可以表示为XML、JSON或RDF，目前在FHIR规范中定义了<%resourcecount%>种不同的资源类型。

本规范描述了一组资源 - 即描述实际可以交换的资源实例的一组资源类型。术语“资源”有时在不明确指明是指类型还是实例的情况下使用 - 使用的上下文会澄清这一点。

示例资源实例

这是一个将患者表示为FHIR对象的示例，使用JSON进行编码。规范中还定义了XML编码。 

{
  "resourceType": "Patient",
  "id" : "23434",
  "meta" : {
    "versionId" : "12",
    "lastUpdated" : "2014-08-18T15:43:30Z"
  }
  "text": {
    "status": "generated",
    "div": "<!-- 省略部分内容 -->"
  },
  "extension": [
    {
      "url": "http://example.org/consent#trials",
      "valueCode": "renal"
    }
  ],
  "identifier": [
    {
      "use": "usual",
      "label": "MRN",
      "system": "http://www.goodhealth.org/identifiers/mrn",
      "value": "123456"
    }
  ],
  "name": [
    {
      "family": "Levin",
      "given": [
        "Henry"
      ],
      "suffix": [
        "第七个"
      ]
    }
  ],
  "gender": {
    "text": "男性"
  },
  "birthDate": "1932-09-24",
  "active": true
}

每个资源实例包括：

  • resourceType（第2行）- 必需：FHIR定义了许多不同类型的资源。请参阅完整索引
  • id（第3行）- 此资源的标识符。在交换资源时始终存在，除了在创建操作（下面）期间
  • meta（第4 - 7行）- 通常存在：所有资源的常用/上下文数据，由基础设施管理。如果没有元数据，则缺失
  • text（第8 - 11行）- 推荐：提供资源的可读的XHTML表示
  • extension（第12 - 17行）- 可选：由可扩展性框架定义的扩展
  • data（第18 - 42行）- 可选：数据元素 - 针对每种资源类型定义了不同的集合

请注意，尽管此规范始终按照定义的顺序显示JSON属性，但许多JSON库会根据其他标准对属性进行排序。

URLs和标识

所有资源都可以有一个标识资源并指定其访问位置的URL。这个URL不会在资源内部表示；它的值在上下文中使用，并且随着资源的复制或其他部署/安全相关的更改而变化。如果通过FHIR RESTful API访问资源（请参见下文），则资源的URL为[base]/[resourceType]/[id]，其中resourceType和id来自资源本身（参见上文）。

一些资源（通常与正式出版周期相关联，而不是与运营医疗保健相关联）在其中具有显式的URL，通常是资源的主要副本的URL。这个URL在资源在系统之间复制时保持不变。有关详细信息，请参见规范URL。

交互操作

对于资源的操作，FHIR提供了一个丰富但简单的REST API，包括以下操作：

  • 创建 = POST https://example.com/base/{resourceType}
  • 读取 = GET https://example.com/base/{resourceType}/{id}
  • 更新 = PUT https://example.com/base/{resourceType}/{id}
  • 部分更新 = PATCH https://example.com/base/{resourceType}/{id}
  • 删除 = DELETE https://example.com/base/{resourceType}/{id}
  • 搜索 = GET https://example.com/base/{resourceType}?search parameters...
  • 历史记录 = GET https://example.com/base/{resourceType}/{id}/_history
  • 事务 = POST https://example.com/base/ (将事务捆绑包提交到系统)
  • 操作 = GET https://example.com/base/{resourceType}/{id}/${opname}

FHIR规范描述了除了这个简单的RESTful API之外的其他类型的交换，包括作为文档的资源组的交换，作为消息的交换，以及使用各种类型的服务的交换。

管理变异性

不同的地缘政治管辖区和医疗行业的不同部分之间存在着广泛的差异，也没有一个中央机构来强制实施共同的业务实践。因此，FHIR规范定义了一个扩展框架并定义了管理变异性的框架。

医疗保健中遇到的变异性的另一个关键方面是，同样的信息可能会被系统中的各方以不同的方式表示，并具有不同的详细程度、粒度和嵌套层次。例如，在某些情况下，血压测量可能只是一个简单的观察，一个生命体征测量，而在其他情况下，它可能是一个包含姿势、运动等控制词汇的丰富的高度定义的数据集。本规范中定义的资源类型侧重于一般的常见用例。更丰富和更具体的内容可以通过在基本资源类型上定义"配置文件"来支持和标准化。

版本管理

在FHIR的上下文中，版本可以指以下三种不同的含义：

  1. FHIR版本：正在使用的FHIR版本是哪个？
  2. 记录版本：用于跟踪资源的更改，并防止更改互相覆盖
  3. 业务版本：以便人们知道他们正在处理的内容的哪个版本（适用于某些类型的资源）

FHIR版本

通常情况下，FHIR版本由上下文确定 - 客户端可以使用CapabilityStatement来了解服务器的信息， 但还有其他方式来管理多个FHIR版本。

记录版本

FHIR服务器不一定要支持版本控制，但强烈建议这样做。 对于FHIR服务器，有三个不同级别的版本控制支持：

  • 不支持版本控制和.meta.version属性（通常也不支持.meta.lastUpdated属性）
  • 支持版本控制和VersionId元属性，但不保留旧版本的历史记录
  • 支持版本控制和VersionId元属性，并提供旧版本的历史记录

此外，服务器可能要求使用版本感知更新，以防止覆盖更改，但此页面上没有描述这一点。

业务版本

某些资源 - 通常是代表经过正式发布周期的内容的资源 - 包含一个明确说明资源表示的内容版本的version元素。 这由人工明确更改，或者根据适用的业务规则由某个自动化过程更改。

创建资源

要创建资源，请向资源类型的相应终点发送HTTP POST请求。 

  POST https://example.com/base/{resourceType}

在下面的示例中，我们看到了创建患者的过程。 

POST /base/Patient HTTP/1.1
Authorization: Bearer 37CC0B0E-C15B-4578-9AC1-D83DCED2B2F9
Accept: application/fhir+json
Content-Type: application/fhir+json
Content-Length: 1198
 
{
  "resourceType": "Patient",
  ...(属性)
}

向服务器提交一个新的患者，并要求服务器使用自己选择的id存储该患者。

注意：

  • /Patient（第1行）- 所有患者的管理器 - 使用资源类型的名称
  • Authorization（第2行）- 参见FHIR安全性
  • Accept, Content-Type（第3-4行）- 所有FHIR资源的内容类型，以JSON表示（或者XML版本的application/fhir+xml）。FHIR资源始终以UTF-8表示
  • id - 客户端不需要为正在创建的资源提供id - 服务器将分配一个。如果客户端分配了一个id，服务器将覆盖它
  • 资源内容，第8行及以后 - 此时没有meta属性。其余部分与上面显示的内容相同

创建响应

响应将包含HTTP代码201，表示资源已成功创建。位置标头指示资源在后续请求中可以获取的位置。服务器可以选择返回一个OperationOutcome资源，但不是必需的。 

HTTP/1.1 201 Created
Content-Length: 161
Content-Type: application/fhir+json
Date: Mon, 18 Aug 2014 01:43:30 GMT
ETag: W/"1"
Location: http://example.com/base/Patient/347
 
{
  "resourceType": "OperationOutcome",
  "text": {
    "status": "generated",
    "div": "<div xmlns=\"http://www.w3.org/1999/xhtml\">操作成功完成</div>"
  }
}

注意：

  • HTTP/1.1 201（第1行）- 操作成功。请注意，强烈建议使用HTTP v 1.1，但不是必需的。
  • ETag（第5行）- 在版本感知更新模式中使用（如果服务器支持版本控制）。
  • Location（第6行）- 服务器分配给资源的ID。URL中的ID必须与后续返回的资源中的ID匹配。
  • OperationOutcome（第9行）- 在此上下文中，OperationOutcome资源没有ID或元数据元素（它们没有受控的身份）。

错误响应

由于各种原因，服务器可能需要返回错误。客户端应该注意与身份验证相关的响应，但是与FHIR内容相关的错误应该使用适当的HTTP状态码返回，并使用OperationOutcome资源提供额外的信息。 以下是服务器因为服务器定义的业务规则而拒绝资源的示例： 

HTTP/1.1 422 Unprocessable Entity
Content-Length: 161
Content-Type: application/fhir+json
Date: Mon, 18 Aug 2014 01:43:30 GMT
 
{
  "resourceType": "OperationOutcome",
  "text": {
    "status": "generated",
    "div": "<div xmlns=\"http://www.w3.org/1999/xhtml\">MRN冲突 - MRN 123456已经分配给另一个患者</div>"
  },
}

注意：

  • 服务器可以使用OperationOutcome的详细信息返回额外的结构化信息

读取请求

通过向所需的资源类型终点发送HTTP GET请求来进行读取资源。 

  GET https://example.com/base/{resourceType}/{id}

这是一个示例。 

GET /base/Patient/347?_format=xml HTTP/1.1
Host: example.com
Accept: application/fhir+xml
Cache-Control: no-cache

注意：

  • 347（第1行）- 正在获取的资源的ID
  • _format=xml（第1行）- 这是客户端指示所需响应格式的另一种方法，除了使用接受头之外，对于无法访问HTTP头的客户端（例如XSLT转换）非常有用（参见Mime类型）
  • 缓存控制（第4行）- 缓存控制很重要，尽管FHIR本身没有提到它 - 参见http://www.w3.org/Protocols/rfc2616/rfc2616-sec13.html或https://www.mnot.net/cache_docs/

读取响应

GET请求的响应包含资源。 

HTTP/1.1 200 OK
Content-Length: 729
Content-Type: application/fhir+xml
Last-Modified: Sun, 17 Aug 2014 15:43:30 GMT
ETag: W/"1"
 
<?xml version="1.0" encoding="UTF-8"?>
<Patient xmlns="http://hl7.org/fhir">
  <id value="347"/>
  <meta>
    <versionId value="1"/>
    <lastUpdated value="2014-08-17T15:43:30Z"/>
  </meta>
  <!-- content as shown above for patient -->  
</Patient>

注意：

  • id（第9行）- 资源的id。这个值必须与读取请求中的id匹配
  • versionId（第11行）- 资源的当前版本id（如果服务器支持版本控制）。最佳实践是该值与ETag匹配（参见版本感知更新），但客户端不能假设这一点
  • 请注意，服务器不需要支持版本控制，但强烈建议这样做
  • lastUpdated（第12行）- 如果存在，这个值必须与HTTP头中的值匹配

搜索请求

除了获取单个已知资源外，还可以通过搜索资源类型终点并使用描述应该检索的资源集合和顺序的一组条件来查找资源集合。一般的模式是： 

  GET https://example.com/base/{resourceType}?criteria

条件是一组HTTP参数，用于指定要返回的资源。搜索操作 

https://example.com/base/MedicationRequest?patient=347

返回上述创建的患者的所有处方。

搜索响应

对搜索请求的响应是一个Bundle：包含一些元数据的匹配资源列表： 

HTTP/1.1 200 OK
Content-Length: 14523
Content-Type: application/fhir+xml
Last-Modified: Sun, 17 Aug 2014 15:49:30 GMT
 
{
  "resourceType": "Bundle",
  "type": "searchset",
  "id" : "eceb4882-5c7e-4ca4-af62-995dfb8cef01"
  "timestamp": "2014-08-19T15:49:30Z",
  "total": "3",
  "link": [
    {
      "relation" : "next",
      "url" : "https://example.com/base/MedicationRequest?patient=347&searchId=ff15fd40-ff71-4b48-b366-09c706bed9d0&page=2"
    }, {
      "relation" : "self",
      "url" : "https://example.com/base/MedicationRequest?patient=347"
    }
  ],
  "entry": [
    {
      "resource" : {
        "resourceType": "MedicationRequest",
        "id" : "3123",
        "meta" : {
          "versionId" : "1",
          "lastUpdated" : "2014-08-16T05:31:17Z"
        }, 
        ... 资源内容 ...
      }, 
    }, 
    ... 2个额外的资源 ....
  ]
}

注意：

  • resourceType/type（第7/8行）- 搜索结果始终是一个类型为“searchset”的bundle
  • id（第9行）- 分配给此特定bundle的标识符。服务器应为此bundle分配一个唯一的id，不会重新使用
  • timestamp（第11行）- （如果服务器支持版本控制）应与HTTP头匹配，并且应该是执行搜索的日期，或者更近，具体取决于服务器如何处理持续更新。时间戳必须与结果中最新的资源的时间戳相同或更近
  • total（第13行）- 搜索结果中的匹配总数。而不是此特定bundle中的匹配数，该bundle可能是结果的分页视图
  • link（第14行）- 一组命名链接，提供与此bundle相关的上下文。在本规范中定义的名称：first，prev，next，last，self
  • entry（第23行）- 此结果集中的实际资源
  • entry.resource.id（第25行）- 请注意，在某些bundle中，资源类型和entry.resource.id的组合必须在bundle中是唯一的
  • 搜索操作还能够返回其他相关资源

更新请求

客户端向服务器发送一个新版本的资源，用于替换现有版本 - 它将其PUT到现有资源的位置： 

  PUT https://example.com/base/{resourceType}/{id}

请注意，{id}处不需要已经存在资源 - 服务器可以选择在指定的地址自动创建资源。 以下是更新患者的示例： 

PUT /base/Patient/347 HTTP/1.1
Host: example.com
Content-Type: application/fhir+json
Content-Length: 1435
Accept: application/fhir+json
If-Match: 1
 
{
  "resourceType": "Patient",
  "id" : "347",
  "meta" : {
    "versionId" : "1",
    "lastUpdated" : "2014-08-18T15:43:30Z"    
  },
  ...
}

注意事项：

  • resourceType（第1行）- URL中的"Patient"必须与资源中的资源类型（第9行）匹配
  • resource id（第1行，"347"）- 这必须与资源中的id（第10行）匹配
  • If-Match（第6行）- 如果提供了此值，它必须与meta.versionId（第12行）中的值匹配，并且服务器必须检查版本的完整性，如果不支持版本，则返回412
  • meta.lastUpdated（第13行）- 此值将被忽略，并由服务器更新（大多数情况下，如果服务器不支持版本控制，则不会更新）
  • resource content（第14行）- 此处未显示，与上面的Patient相同

更新响应

对于更新请求的响应包含元数据/状态，以及可选的操作结果（OperationOutcome）： 

HTTP/1.1 200 OK
Content-Length: 161
Content-Type: application/fhir+json
Date: Mon, 18 Aug 2014 01:43:30 GMT
ETag: W/"2"
Location: https://example.com/base/Patient/347/_history/2
 
{
  "resourceType": "OperationOutcome",
  "text": {
    "status": "generated",
    "div": "<div xmlns=\"http://www.w3.org/1999/xhtml\">操作成功完成</div>"
  }
}

注意：

  • ETag（第5行）- 这是新版本的versionId，并且也可以在位置头部（第6行）找到（如果服务器支持版本控制）

基本资源内容

这是一个示例，展示了所有资源中的所有信息，完全填充： 

{
  "resourceType" : "X",
  "id" : "12",
  "meta" : {
    "versionId" : "12",
    "lastUpdated" : "2014-08-18T15:43:30Z",
    "profile" : ["http://example-consortium.org/fhir/profile/patient"],
    "security" : [{
      "system" : "http://terminology.hl7.org/CodeSystem/v3-ActCode",
      "code" : "EMP"
    }],
    "tag" : [{
      "system" : "http://example.com/codes/workflow",
      "code" : "needs-review"
    }]
  },
  "implicitRules" : "http://example-consortium.org/fhir/ehr-plugins",
  "language" : "X"
}

实施者注意事项：

  • resourceType（第2行）- 在每个资源中始终存在。在XML中，这是资源的根元素的名称
  • id（第3行）- 在创建资源时定义，并且永远不会更改。只有在资源首次创建时才会缺失
  • meta.versionId（第5行）- 每当资源内容发生更改时更改（除了meta中的最后3个元素 - profile，security和tag）
  • meta.lastUpdated（第6行）- 当versionId更改时发生更改。不支持版本的系统通常也不跟踪lastUpdated
  • meta.profile（第7行）- 断言内容符合某个配置文件。有关详细讨论，请参见扩展和限制资源。可以根据配置文件和值集的更改或系统重新检查一致性来更改
  • meta.security（第8 - 11行）- 应用于此资源的安全标签。这些标签以特定方式将资源连接到整体安全策略和基础设施。安全标签可以在资源更改时更新，或者在安全子系统选择时更新
  • meta.tag（第12 - 16行）- 应用于此资源的标签。标签用于将资源与流程和工作流程相关联。在解释资源的含义时，应用程序不需要考虑这些标签
  • implicitRules（第17行）- 表示关于如何安全处理资源的自定义协议。不鼓励使用此功能，因为它限制了共享，但有时是必要的
  • language（第18行）- 资源的基本语言。资源允许包含其他语言的内容；这只是基本语言，但应该是资源的主要语言

所有资源的基本属性在资源类型Resource和DomainResource上定义。
